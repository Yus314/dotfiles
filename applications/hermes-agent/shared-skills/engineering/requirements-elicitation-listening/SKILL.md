# Requirements Elicitation Listening

## When to Use

Use this skill when the user wants to talk through an idea, organize requirements,
or be listened to before receiving a plan. It also applies to reflective career or
life-direction elicitation when the user wants one-question-at-a-time dialogue
rather than immediate advice.

## Core Stance

1. Start as a reflective listener, not a solution architect.
2. Mirror the user's terms and metaphors; preserve their conceptual distinctions.
3. Summarize in small increments and invite continuation.
4. Do not prematurely collapse an idea into a familiar product or technical category.
5. When the user corrects a framing, explicitly update the working definition and
   do not re-center the discarded frame.
6. Ask exactly one high-leverage question at a time when requested; do not turn the
   conversation into a questionnaire.
7. Treat reflections as hypotheses, not conclusions.

## Product and Software Concepts

Maintain a lightweight running model:

- Purpose
- Core metaphor
- Target users
- What is unified
- Explicit non-goals
- Examples and whether they are central or illustrative
- Desired first demo or MVP
- Risks and failure modes
- Open questions

Listen until the object-level concept is stable before proposing architecture,
implementation details, or a product category.

## Reflective Career or Life-Direction Variant

Elicit the user's values, energy sources, desired impact, preferred work process,
learning conditions, environment needs, and pain signals before connecting them to
job titles or concrete plans. See
[`references/career-reflection-elicitation.md`](references/career-reflection-elicitation.md)
for a generic question sequence.

## Useful Moves

- Restate the current understanding in the user's vocabulary.
- Separate goals, non-goals, examples, design principles, user value, and possible
  implementation.
- Mark examples as examples when they are not the center of the concept.
- Briefly name what became clearer before asking the next question.
- If asked whether the dialogue is helping, answer explicitly and summarize the
  current understanding.

## Pitfalls

- Overemphasizing a compelling example and making it the center.
- Asking many questions before the user has finished laying out the idea.
- Jumping to implementation details or job-title recommendations too early.
- Treating a user's need for initial scaffolding as lack of long-term autonomy.
- Storing session-specific concepts or personal profiles in this shared package;
  those belong in profile-local notes or adapters.
